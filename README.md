# PawSwipe — Swipe-to-Vote Adoptable Pets App

> CMPE 285 Final — Question 10 Submission

---

## Theme

**Adoptable Pets** — Users swipe through 100 real-breed adoptable animal profiles (dogs, cats, rabbits, birds, hamsters, guinea pigs) and vote yes (want to adopt) or no (skip). A results view shows aggregate vote counts across all users.

---

## How to Run

### Prerequisites
- Python 3.9+
- pip

### 1 — Install & run

```bash
cd backend
pip3 install -r requirements.txt
python3 server.py
# → App at http://localhost:5050 (API + frontend)
```

Open **http://localhost:5050** in a browser (desktop or phone on same WiFi — use your machine’s LAN IP instead of `localhost` on mobile).

> Alternatively, open `frontend/index.html` directly; the API must still be running at `http://localhost:5050`.

## Architecture

**Frontend:** A single-file vanilla JS + HTML5 application served statically. No build toolchain needed. Touch/mouse drag events are handled directly with pointer coordinates to produce the swipe-card feel. Card tilt, color hints (green/red), and progress tracking are all CSS + JS. Session identity is stored in `sessionStorage` so one user's votes persist across page reloads within a session.

**Backend:** Python Flask server with a single SQLite database (`votes.db`, auto-created on first run). The pets list is loaded from `pets.json` at startup (generated by `seed.py`). The `/vote` endpoint uses a `UNIQUE(session_id, item_id)` constraint to enforce idempotency, a second vote on the same item updates rather than double-counts. The `/results` endpoint aggregates via a single SQL GROUP BY query. Usernames are stored in a `users` table; `/analytics` reports global and per-session stats including average decision time (ms), sent with each vote from the client.

**Trade-offs under time pressure:**
- SQLite over Postgres: zero setup, single file, more than sufficient for a class demo. Would swap to Postgres for concurrent write load.
- No auth / magic-link: sessionStorage UUID is anonymous but simple. Adequate for the exercise; a real app would use JWT.
- Static JSON seed: `pets.json` is loaded once at startup. An admin route to add items was skipped for time.

- [x] 100 distinct items (100 pet profiles across 6 species/65+ breeds)
- [x] Swipe right = yes, swipe left = no
- [x] Visual feedback: card tilt, green/red colour hint, threshold indicator
- [x] Smooth transition to next card after vote
- [x] Results view (Results button, **swipe down** on card area, or end-of-deck prompt)
- [x] Aggregate yes/no counts across all users, sortable (Most Loved, Most Voted, Divisive, Least Voted)
- [x] Backend with SQLite persistence — votes aggregate across users
- [x] Idempotency: UNIQUE constraint prevents double-counting
- [x] End-of-deck state handled gracefully

## Stretch Goals Completed

- [x] Lightweight sign-in (username only, no password) — `POST /auth/signin`; each sign-in gets a new session and starts at pet #1 (results/analytics stay global)
- [x] Anonymous session identity (sessionStorage UUID) — votes remembered across reloads
- [x] Previous votes resumed on reload (fetches session votes from backend)
- [x] Undo last swipe (↩ button + `DELETE /vote`)
- [x] Matches view — pets you voted yes on with global yes-rate ≥ 60%
- [x] Real-time results refresh (15s polling while results view is open)
- [x] Analytics page — total swipes, voters, signed-in users, avg decision time (`GET /analytics`)
